# fTree 支持与反馈

fTree 统一通过 GitHub 接收、分类和追踪反馈。请先判断事项类型，再选择对应入口；分类正确、描述完整可以显著缩短确认时间。

## 支持入口

| 类型 | 适用范围 | 提交入口 | 追踪列表 |
| --- | --- | --- | --- |
| 新需求 | 新功能、体验改进、新云服务、流程优化 | [提交需求 Issue](https://github.com/mulspace/ftree/issues/new?template=feature_request.yml) | [需求列表](https://github.com/mulspace/ftree/wiki/需求列表) |
| 问题 | 页面错误、数据异常、同步失败、无法恢复、功能与说明不一致 | [提交问题 Issue](https://github.com/mulspace/ftree/issues/new?template=bug_report.yml) | [问题列表](https://github.com/mulspace/ftree/wiki/问题列表) |
| 其他 | 使用咨询、资料整理方法、文档讨论、非故障类交流 | [在 Wiki 登记](https://github.com/mulspace/ftree/wiki/其他列表) | [其他列表](https://github.com/mulspace/ftree/wiki/其他列表) |

不要用私信、普通邮件、电话或其他渠道替代上述入口。公开记录便于用户确认是否已有相同事项，也便于作者持续更新状态。

## 新需求填写要求

需求 Issue 必须说明：

1. 背景与痛点：当前在什么场景遇到了什么限制。
2. 目标：希望解决的问题，而不仅是指定某个按钮或技术方案。
3. 使用场景：谁在何时、按什么流程使用。
4. 期望结果：完成后应看到什么可验证结果。
5. 范围边界：哪些内容属于本需求，哪些不属于。
6. 替代方案：目前如何绕过，为什么仍不满足。
7. 数据与隐私影响：是否新增个人资料、第三方服务、权限或保存位置。
8. 优先程度和理由。

一个 Issue 只描述一个独立需求。不要把多个无关功能合并在同一 Issue，也不要只写“希望增加某功能”而没有场景和验收标准。

## 问题填写要求

问题 Issue 必须说明：

1. 问题摘要：用一句话描述异常。
2. 环境：浏览器、版本、操作系统和访问地址。
3. 前置条件：使用的页面、家谱状态、是否刚执行同步或恢复。
4. 完整复现步骤：从打开页面开始，按编号写清每一步。
5. 实际结果：实际发生了什么。
6. 期望结果：正常情况下应该怎样。
7. 出现频率：必现、偶发或仅发生一次。
8. 影响范围：是否阻断使用、是否可能影响本地或云端数据。
9. 错误文字或脱敏截图。
10. 已尝试的处理方式及结果。

提交前请先下载当前 `.tree` 备份；发生数据覆盖风险时，停止继续重复操作。

## 其他问题填写要求

其他事项在 Wiki“其他列表”中新增一行，至少写明：

- 日期和 GitHub 用户名。
- 分类和简短标题。
- 背景、具体问题及希望得到的答复。
- 已查阅的文档和已尝试的方法。
- 当前状态：待回复、处理中、已答复或已关闭。

如果讨论后确认属于新需求或软件问题，应转为对应 Issue，并在 Wiki 条目中补充 Issue 链接。

## 隐私和安全要求

任何公开反馈都不得包含：

- 真实 `.tree` 文件或其可下载链接。
- Secret Gist URL。
- Cookie、OAuth code、Access Token、Refresh Token。
- Client Secret、服务密钥或完整回调参数。
- 完整成员名单、私人备注、证件资料。
- 在世人员精确 GPS 或住宅地址。

可以使用虚构人物、最小复现数据和脱敏截图。若意外提交敏感信息，应立即删除内容、撤销相关凭据，并通知仓库维护者处理历史缓存。

## 状态追踪

- 需求和问题以 GitHub Issue 状态为准：Open 表示仍需处理，Closed 表示已完成、无法复现、暂不采纳或已给出结论。
- 需求 Issue 标题统一以 `[需求]` 开头。
- 问题 Issue 标题统一以 `[问题]` 开头。
- 其他事项由 Wiki“其他列表”的状态列追踪。
- 重复事项应链接到已有记录，不要平行创建多个相同条目。
