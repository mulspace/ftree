# fTree 数据操守与操作手册

本手册同时适用于家谱资料整理者、家庭管理员、协作负责人和站点维护人员。目标是在“能用”之外，建立可持续、可审计、尊重在世人员的资料管理习惯。

## 1. 基本原则

### 1.1 尊重与知情

- 在世人员资料应在合理知情和同意基础上录入。
- 不因家谱整理目的收集与亲属关系无关的敏感信息。
- 对领养、再婚、非婚生、医疗、财务和家庭争议等事项保持克制。
- 资料存在分歧时记录来源和不确定性，不把推测写成事实。

### 1.2 最少收集

- 姓名、性别和必要亲属关系通常足以完成基础家谱。
- GPS 优先记录祖居、祠堂、墓地或村镇级位置，不记录在世人员住所门牌。
- 备注不记录密码、完整身份证号、银行卡、健康诊断或未经证实的评价。
- 导入旧资料后，应检查是否混入不应继续保存的字段。

### 1.3 分级共享

建议把资料分为：

1. 公开：历史人物、公开史料和已获授权的概览。
2. 家庭内部：在世人员关系、一般备注和非精确位置。
3. 严格限制：精确 GPS、争议信息、证件资料和其他敏感内容。

公开图片只应包含第一类信息。`.tree` 原始档案默认按第三类保护。

### 1.4 可恢复性

任何重要修改前必须存在可用备份。没有经过恢复验证的文件不能视为可靠备份。

## 2. 角色与责任

### 2.1 资料整理者

- 核实来源。
- 按统一格式录入姓名和关系。
- 修改前下载备份。
- 标记不确定资料。
- 不擅自公开在世人员信息。

### 2.2 家庭资料负责人

- 制定录入范围、命名规则和保留周期。
- 决定谁可以接触完整档案。
- 定期检查云端授权与备份。
- 处理更正、删除和争议请求。

### 2.3 站点维护人员

- 不索取用户真实 `.tree`、OAuth Token 或 Cookie。
- 不把生产 Secret 写入源代码、日志、工单或聊天。
- 权限按最少需要配置。
- 发布前执行测试、隐私检查和回退准备。
- 安全事件发生后及时轮换凭据并说明影响。

## 3. 日常操作标准

### 3.1 开始工作前

1. 确认使用正确的浏览器配置文件和设备账户。
2. 确认不是无痕窗口。
3. 下载当前 `.tree` 并记录时间。
4. 检查本轮资料来源和授权范围。
5. 明确本轮修改目标，避免无计划批量更改。

### 3.2 录入过程中

- 每完成一个家庭单元，核对父母组合、配偶和子女。
- 同名人员必须借助上下代、性别和来源进一步确认。
- 不确定关系先保留在离线研究笔记，不直接连接到正式家谱。
- 大批量修改分阶段进行，每阶段单独备份。
- 不在公共直播、会议共享屏幕或录屏中展示私人资料。

### 3.3 工作结束后

1. 检查家谱数量和成员数量是否符合预期。
2. 随机抽查至少三个修改过的家庭单元。
3. 下载带日期的新 `.tree`。
4. 如启用云同步，确认上传方向和目标账户。
5. 在维护记录中写明日期、范围、资料来源和操作人。
6. 锁屏或退出公共设备。

## 4. 命名与资料质量

- 姓名使用家族约定的标准字形；异体字可写入备注。
- 不用“某某父亲”“未知妻子”等临时名称冒充真实人物。
- 同一日期格式保持一致。
- 版本号建议使用 `YYYY.MM` 或语义版本。
- 堂号应简洁，避免加入私人联系方式。
- 修改争议资料时保留来源说明，但来源中不要包含不必要的个人联系方式。

## 5. 关系修改规范

### 新增父母、配偶或子女

- 至少由一种可靠来源支持。
- 添加后从相关成员两侧检查关系是否一致。
- 多配偶、多父母等复杂情况应逐组录入并验证布局。

### 删除成员

- 删除前查看受影响子女和配偶关系。
- 下载删除前备份。
- 确认该成员没有承担跨家谱关联。
- 删除后检查是否出现孤立成员或错误父母组合。

### 跨家谱关联

- 只关联确定为同一人的同名、同性别成员。
- 不因为姓名相同就建立关联。
- 关联后测试溯源、继承和只读保护。
- 删除关联前评估两个家谱的展示影响。

## 6. 备份制度

推荐采用 3-2-1 原则：

- 至少三份副本。
- 至少两种不同介质。
- 至少一份离线或异地保存。

建议结构：

- 浏览器本地工作副本。
- 个人云服务中的同步副本。
- 加密磁盘或离线移动介质中的日期备份。

每月至少执行一次恢复演练：

1. 在独立浏览器配置文件或测试设备打开网站。
2. 导入最近备份。
3. 核对家谱数、成员数、主要关系和关联。
4. 记录验证结果。
5. 测试环境结束后清除其中的私人资料。

## 7. 云同步操作纪律

- 授权前确认弹窗域名属于 Google、GitHub 或 GitLab。
- 确认选中的是预期账户，尤其是同时登录多个账户时。
- 上传前下载本地备份。
- 下载前下载当前本地备份。
- 不在网络不稳定时反复点击同步。
- 同步失败后先检查目标账户，不盲目重复覆盖。
- 退出授权后，单独检查云端文件是否需要保留或删除。

对于 GitHub，Secret Gist 不视为严格私有。不得转发 Gist URL，不得把它写入公开 Issue、Wiki、截图或日志。

## 8. 分享规范

### 分享图片

- 隐去在世人员精确位置、敏感备注和不必要关系。
- 只向明确接收人发送。
- 提醒接收人不要二次公开传播。
- 社交平台发布前再次检查图中所有在世成员。

### 分享 `.tree`

- 原则上不通过公共群、公开链接或普通邮件附件发送。
- 使用受控的一对一渠道和短期有效链接。
- 接收人应明确保管责任和删除时间。
- 分享后记录接收人、时间、目的和文件版本。

## 9. 支持人员操守

处理用户问题时：

- 先判断属于新需求、软件问题还是其他事项，并引导到对应 GitHub 入口。
- 要求需求写清背景、目标、使用场景、验收标准、范围和隐私影响。
- 要求问题写清环境、前置条件、复现步骤、实际结果、期望结果、频率和影响。
- 优先索取复现步骤、浏览器版本和脱敏截图。
- 不索取完整备份来解决一般界面问题。
- 不要求用户发送 Cookie、Token、OAuth 回调 URL 或 Secret。
- 必须使用数据样例时，使用虚构人物和虚构关系。
- 意外收到私人档案时，不转发、不复制、不上传其他服务，并尽快通知发送者安全删除。
- 公共问题记录中只保留经过脱敏的技术信息。

## 10. 站点发布检查表

发布前：

- [ ] 构建和测试全部通过。
- [ ] OAuth 回调地址与正式域名一致。
- [ ] 没有 `.dev.vars`、Token、Cookie 或 Secret 进入提交。
- [ ] 没有真实 `.tree` 或家谱截图进入发布包。
- [ ] 云存储权限没有超出功能所需。
- [ ] 上传、下载、退出和错误处理均已验证。
- [ ] 隐私提示与实际代码行为一致。
- [ ] 已准备上一版本和回退步骤。
- [ ] 已记录发布人、时间和变更摘要。

发布后：

- [ ] 正式域名 HTTPS 正常。
- [ ] 静态页面可打开。
- [ ] 三个 OAuth 回调均返回预期结果。
- [ ] 使用测试账户验证上传与下载。
- [ ] 确认测试数据已从第三方账户清理。
- [ ] Wiki 和操作手册链接可访问。

## 11. Secret 管理

- Client Secret 和 Cookie 加密密钥只存放在正式 Secret 管理中。
- 本地开发值只写入被 Git 忽略的开发变量文件。
- 不用生产账户和生产 Token 做自动化测试。
- 人员离岗、设备遗失或疑似泄露时立即轮换。
- 轮换 Cookie 加密密钥会使现有授权 Cookie 失效，应提前准备用户通知。
- 轮换 OAuth Client Secret 后，应重新验证所有回调。

## 12. 安全事件处置

### 12.1 档案或链接泄露

1. 立即停止继续传播。
2. 删除公开帖子、附件和共享链接。
3. 删除或替换云端副本。
4. 判断是否包含在世人员、GPS 或敏感备注。
5. 通知受影响的资料负责人。
6. 从泄露前可信备份恢复，并保存最少必要的事件记录。

### 12.2 OAuth 凭据泄露

1. 在第三方平台撤销 Token 或应用授权。
2. 轮换相关 Client Secret。
3. 必要时轮换 Cookie 加密密钥，使全部会话失效。
4. 检查 Gist、Snippet 和应用数据是否被修改。
5. 通过可信渠道发布重新授权说明。

### 12.3 设备丢失

1. 远程锁定或擦除设备。
2. 修改设备账户和第三方云账户密码。
3. 撤销该设备相关会话和 OAuth 授权。
4. 检查云端备份完整性。
5. 从可信备份在新设备恢复。
